Dimension 01

Vision — from concept to category claim

The original problem

The vision was beautiful but unownable. "AI-native OS" is aspirational but undefended — anyone can say it. There was no proof that multi-agent is better than single-agent, no unique insight about why now, and no crisp answer to "why can't OpenAI just build this?"

The upgraded vision: own the kernel layer

The insight nobody has said clearly yet: model providers are vertically integrated toward the end-user and will never build a neutral orchestration kernel. OpenAI Operator works for OpenAI models. Claude Computer Use works for Anthropic models. Google Mariner works for Gemini.

AgentOS is the model-neutral runtime — the layer underneath every agent, regardless of who made the model. Like Linux vs Windows Server: the kernel doesn't care who made the application. This framing is defensible, accurate, and positions AgentOS as infrastructure rather than a product competing on features.

The one-sentence pitch (rewritten)

Before (unownable)

"AgentOS is an AI-native operating environment that makes your computer behave like an intelligent workforce."

After (category-defining)

"AgentOS is the model-neutral agent kernel for desktop — the operating layer that lets any AI model control any computer, at any cost, without vendor lock-in."

Three strategic pillars that replace the wishlist

Pillar 1

Local-first execution

Privacy and cost control no cloud-native competitor can match. Local models via Ollama, local memory, local execution. Enterprise CISO approves in 30 minutes because nothing leaves the machine. This is structurally impossible for OpenAI Operator.

Pillar 2

Model neutrality

One workspace. Six different model providers. Route each task to the cheapest/best model for that task type. No other product does this with a kernel-level scheduler. This is the Kubernetes of AI agents.

Pillar 3

Developer ecosystem

Open kernel API from day one. Third-party agent packs are the equivalent of VS Code extensions. Network effects compound. Model providers cannot replicate this without alienating their developer base.

The one thing that makes this defensible

The Agent Pack ecosystem is the moat. When 500 developers have published packs for AgentOS, switching costs are real. Build the ecosystem before the model providers notice it matters. Target: 100 published packs before Series A.

What to remove from the spec permanently

  • Agent "personalities" as a property system — this is a system prompt, not an architecture

The one UX principle that changes everything

Every screen must answer: "what is happening right now, what did it cost, and what should I do next?" Agent UX that doesn't answer these three questions in 3 seconds will be abandoned. Every tab, every panel, every agent card must be designed around this triad — not around the agent's internal state.

Dimension 03

Architecture — production-grade from day one

The original problem

The spec described the Agent Kernel without specifying the language, the concurrency model, the failure recovery strategy, the IPC mechanism, or the sandboxing approach. "Central orchestration system" is not an architecture. Python async loops will collapse under production load on a desktop. Computer control on heterogeneous hardware is not mentioned as hard.

Mandatory technology decisions

Agent kernel process — write in Rust
Tokio async runtime Actor model (Actix) Message passing, no shared state Graceful agent death + restart Resource quotas per agent

Why Rust: Python async fails under CPU contention on desktop. Rust gives you zero-cost concurrency, OS thread control, and memory safety without a GC pausing at the worst moment.

Desktop shell — Tauri (not Electron)
React frontend WebSocket to Rust kernel OS keychain for secrets React Flow for visual builder 30MB binary vs 200MB Electron

Tauri uses the system WebView — faster startup, smaller bundle, better OS integration. Critical for security: native keychain access, OS-level sandboxing.

Memory system — three explicit tiers
T1: In-context (conversation window) T2: Session (LanceDB local vector) T3: Long-term (SQLite structured) ACL: per-agent read/write scopes Source tagging: trusted vs untrusted

Every memory entry carries: source URL/agent, trust level, timestamp, and a content hash. Untrusted content (web scrapes) cannot be promoted to T3 without explicit user confirmation.

Computer control — staged rollout
Stage 1: Playwright (browser only) Stage 2: Filesystem with path allowlist Stage 3: Terminal with command allowlist Stage 4: Native apps via accessibility API Dry-run mode for all destructive ops

Never ship all four stages simultaneously. Browser automation (Stage 1) alone is a shippable product. Each new stage adds attack surface and fragility — earn the right to add it.

Plugin sandbox — WASM mandatory
Wasmtime runtime Capability manifest (network egress allowlist) Code signing required No plugin gets filesystem access by default API rate limit per plugin

Every Agent Pack runs in WASM. It cannot access memory it wasn't given. It cannot open network connections not declared in its manifest. This is non-negotiable for a marketplace.

Model routing engine
Task classifier (local, fast, tiny model) Route by: complexity, cost ceiling, privacy flag Fallback chain: cloud → local → error Latency SLO per task type Cost ceiling per workspace per day

The hardest engineering problem nobody is talking about

Agent-to-agent communication consistency. When Agent A tells Agent B to do something and B fails halfway through, who owns recovery? What is the rollback? What does the user see? You need a distributed transaction protocol between agents — think two-phase commit but for LLM calls. Nobody has designed this cleanly. Doing it well is your technical moat.

Dimension 06

Security — the architecture of trust

The original problem

The spec described agents with filesystem access, terminal access, email access, and browser access — with no mention of how any of these would be sandboxed, audited, or constrained. This is a security nightmare that would produce the first serious AI agent breach incident within 6 months of shipping.

Security architecture — non-negotiables before shipping

Principle of least privilege — enforced at the kernel

Every agent declares its capability requirements at startup. The kernel grants the minimum required. An agent asking for "filesystem write" must specify the exact path subtree. An agent asking for "network" must specify the hostnames. No blanket capabilities. No capability inheritance from parent to child agent without explicit user approval.

Prompt injection firewall

All content read from external sources (web, files, emails) passes through an injection classifier before being inserted into any agent context. The classifier runs locally (small fine-tuned model). Content flagged as potentially adversarial is shown to the user with a warning before being passed to an agent. Never trust content from outside the user's control as instructions.

Immutable action log

Every action an agent takes — every file write, every terminal command, every API call — is logged to an append-only local log before it executes. Users can audit and replay. Destructive operations (file delete, email send, payment initiation) require a user confirmation step and are highlighted in the log. This is the "undo" mechanic that makes autonomous agents trustworthy.

Secrets never touch disk unencrypted

API keys, OAuth tokens, and credentials are stored exclusively in the OS keychain (Windows Credential Manager / macOS Keychain / Linux Secret Service). They are decrypted in memory for the duration of a single API call and zeroed afterward. The Agent Kernel has no persistent access to credentials — it requests them per call from the keychain daemon.

WASM plugin isolation

All Agent Packs run in Wasmtime. They cannot escape the sandbox. They have no default capabilities. Every capability (network call to a specific host, reading a specific environment variable) must be declared in the pack manifest and approved by the user at install time — similar to Android app permissions. A compromised plugin cannot access other agents' memory, the user's filesystem, or stored credentials.

The security advantage vs cloud competitors

Because AgentOS runs locally, you have a security story that no cloud-native competitor can match: your data never leaves your machine. Agents use local models for sensitive tasks. Memory is encrypted at rest on your own disk. The kernel has no external telemetry by default. For any company with a CISO, this is the deciding factor over Manus or Operator.

Dimension 08

MVP — the 10-week build that proves everything

The original problem

The spec had no MVP. It described the full vision as if it would be built simultaneously. An investor reading this sees "team that doesn't know how to ship." The test for a good MVP: if this is all we ever shipped, would it teach us the one thing we need to know?

The single question the MVP must answer

Does a multi-agent team produce measurably better code than a single model when given the same spec? If yes, you have a category. If no, you need a different thesis.

What to fake in the MVP

Dimension 09

Investor narrative — the pitch that gets a term sheet

The original problem

The original spec had no investor narrative. It was a product spec, not a company argument. Investors need: a market insight that is non-obvious, a reason why you win and OpenAI doesn't, evidence that the team can execute, and a specific bet on timing. None of these were present.

The four-sentence pitch

Opening

"Every major AI model provider is racing to build a computer-controlling agent. OpenAI built Operator for OpenAI models. Anthropic built Computer Use for Claude. Google built Mariner for Gemini. None of them will ever build the neutral layer — because neutrality conflicts with their core business model. We are building that layer: AgentOS, a model-neutral agent kernel for desktop, with local execution, an open API, and a developer ecosystem that compounds."

The traction narrative that matters

Do not pitch users or signups. Pitch evidence that multi-agent is better. One reproducible benchmark — "when given the same spec, our 4-agent team produces code that passes 73% more test cases than a single GPT-4o call, at 2.3x the cost but 4x the quality" — is worth more than 5,000 waitlist signups. This benchmark is the company's intellectual property. It validates the thesis. It generates press. It answers the "why multi-agent?" question permanently.
